请你介绍一下 CSS 的两种盒模型

面试问答：

回答：

CSS 的盒模型分为两种：标准盒模型、IE 盒模型 。标准盒模型从外到内包括 padding、border、margin、content。IE 盒模型也是这样的构造。

追问：

在你的 BI 大屏开发中，使用的是标准盒模型还是 IE 盒模型？

回答：

标准盒模型

标准回答：

  1. 标准盒模型：标准盒模型的 width 和 height 只包括 content
  2. IE 盒模型：IE 盒模型的 width 和 height 属性包框 margin、border、padding

请你介绍一下 Vue3 中的响应式数据的原理

面试回答：

（Vue3 中我以为面试官问的响应式数据是 ref 和 reactive，但是因为我记不太清楚，我感觉可能是数据拦截，所以答的没有什么底气）

Vue3 的响应式数据主要是通过数据拦截来实现的，数据拦截他有 13 种方法，可以准确的拦截对象所有的变化，而不像 Vue2 一样，不能完整拦截对象，需要层层便利，造成性能浪费。

标准回答：

  1. Vue3 的响应式系统是基于 Proxy（数据拦截） 实现的
  2. 当我们使用 reactive 或 ref 创建响应式数据的时候，Vue 会通过 Proxy 对对象进行代理
  3. 当组件渲染时，访问响应式数据会使用 get，Vue 会对依赖进行收集，记录组件渲染依赖那些数据
  4. 当数据发生改变时，会调用 set，Vue 会通知相关的依赖重新执行，从而触发重新渲染，实现视图更新
  5. 相比 Vue2，Vue3 的 Proxy 属性还可以监听对象的增加、删除以及数组变化，性能和功能更加完善

如果有一个图片更新了，需要通知其他组件同步更新，你会怎么做？ 至少写出三种方法

面试回答：

（当时没有理解面试官是在问组件通信方式，以为是在问图片更新的具体实现逻辑，因此没有回答到组件通信方向上，往子传父的方向想了）

  1. 子组件跑出 emit，父组件接受到 emit 调用 handle 的函数
  2. 通过 Pinia 进行通行…

标准回答：

  1. Props（父传子）

父组件将图片地址通过 props 传递给子组件。

当父组件中的图片地址发生变化时，子组件会自动接收到最新数据并重新渲染。

  1. Emit（子传父）

如果图片上传功能在子组件中，可以通过 emit 通知父组件。

父组件接收到事件后更新图片地址。

  1. Pinia（跨组件通信）

如果页面中多个组件都依赖同一张图片（如头像），可以将图片地址存储到 Pinia 中。

修改 Pinia 中的数据后，所有使用该状态的组件都会自动更新。

  1. Ref + defineExpose

父组件通过 ref 获取子组件实例。

子组件通过 defineExpose 暴露方法。

父组件可以主动调用子组件方法实现刷新

请问在 BI 大屏开发中，如果遇到大量数据，你会如何处理？

标准回答

在 BI 大屏开发中，大量数据主要会带来两个问题：

  1. 数据请求压力
  2. 页面渲染性能问题

针对不同场景可以采用以下方案：

  1. 后端分页

如果是表格数据：

10万条数据
↓
后端分页
↓
每次返回20~50条

避免一次性加载全部数据。

  1. 虚拟列表

如果是：

  • 订单列表
  • 用户列表
  • 日志列表

可以采用虚拟滚动。

原理：

10000条数据
↓
只渲染可视区域
↓
实际DOM可能只有20个

减少DOM数量。

  1. 数据聚合

对于图表：

10万条原始数据
↓
按天统计
↓
生成汇总数据

避免图表直接渲染全部数据点。

  1. ECharts 数据采样

对于折线图：

100000个点
↓
采样
↓
5000个点

降低绘图压力。

  1. 懒加载

对于图片或模块：

进入视口
↓
再加载

减少首屏压力。

  1. 防抖与节流

对于：

  • 搜索框
  • 数据筛选
  • 实时查询

避免频繁发送请求。

  1. Web Worker

如果涉及：

  • 大量统计计算
  • Excel解析
  • 数据分析

可以将计算放入 Web Worker。

避免阻塞主线程。

总结：

BI 大屏中面对大量数据时，应根据场景选择不同优化方案：

  • 数据过多 → 分页
  • DOM过多 → 虚拟列表
  • 图表数据过多 → 聚合、采样
  • 请求过多 → 防抖节流
  • 计算过多 → Web Worker

请说出下列代码的打印结果

console.log(1);

setTimeout(() => {
  console.log(2);
}, 0);

new Promise((resolve) => {
  console.log(3);
  resolve();
}).then(() => {
  console.log(4);
});

console.log(5);

面试回答：

打印结果：1 5 3 4 2

解释：首先先执行同步代码，console.log(1) 和 console.log(5)，接着执行 promise，console.log(3) console.log(4)，最后执行 console.log(2)

标准答案：

1 3 5 4 2

执行流程拆解

  1. 同步代码（主线程立即执行）

console.log(1)
console.log(3)
console.log(5)

输出：
1 → 3 → 5

  1. 微任务（Microtask）

Promise.then(() => {
console.log(4)
})

输出：
4

  1. 宏任务（Macrotask）

setTimeout(() => {
console.log(2)
}, 0)

输出：
2

错误点：

误以为 Promise 是整体异步 ， Promise executor 是同步执行 ， then 才是微任务 。

总结：

  1. 面试八股不够熟悉
  2. 应用场景不够熟悉，导致没有理解到组件通行的题目
  3. 很多知识只停留在了解的层面，不能很流畅顺利的表达
